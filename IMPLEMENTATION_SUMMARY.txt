KAJVE API-MOBILE - MICROSERVICIO COMPLETADO
==========================================

RESUMEN DE GENERACIÓN
=====================

Se ha generado exitosamente el microservicio "api-mobile" para ser consumido por la app Flutter del Productor,
siguiendo arquitectura Clean Architecture y buenas prácticas de Go.

CARACTERÍSTICAS IMPLEMENTADAS
=============================

✓ ARQUITECTURA:
  - Clean Architecture con capas: domain/, application/, infrastructure/, delivery/
  - Sin frameworks pesados, net/http estándar + chi router
  - Patrón Ports & Adapters (interfaces)
  - Inyección de dependencias manual (sin contenedor)

✓ AUTENTICACIÓN (COMPLETA):
  - POST /auth/login → JWT access + refresh token
  - POST /auth/refresh → Renovación de tokens
  - Middleware JWT validando en todas las rutas protegidas
  - Tokens: HS256 con 24h expiracion (access) y 30d (refresh)
  - Contraseñas: bcrypt

✓ DISPOSITIVOS (COMPLETA):
  - POST /devices/link → Vinculación de ESP32 con provisioning token
  - Validación de token de un solo uso (SHA256)
  - Creación automática de lotes
  - RLS automático después de vinculación

✓ LOTES:
  - GET /lotes → Lista de lotes del usuario
  - GET /lotes/{id} → Detalle del lote
  - POST /lotes → Crear nuevo lote

✓ LECTURAS AMBIENTALES:
  - GET /lotes/{id}/lecturas → Últimas N lecturas con limit configurable

✓ ALERTAS:
  - GET /lotes/{id}/alertas → Alertas del lote

✓ PREDICCIONES:
  - GET /lotes/{id}/predicciones → Predicciones generadas por ml-service

✓ RECOMENDACIONES:
  - GET /lotes/{id}/recomendaciones → Recomendaciones

✓ HISTORIAL:
  - GET /lotes/{id}/historial → Historial de eventos

✓ REPORTES:
  - POST /reportes → Solicitar generación de reporte
  - Estado: pendiente → procesando → completado

✓ SEGURIDAD OBLIGATORIA:
  - JWT validado en middleware (Bearer token)
  - Rate limiting: 100 req/min por IP con golang.org/x/time/rate
  - RLS de PostgreSQL activado (SET app.current_user_id)
  - Validación de inputs con go-playground/validator
  - CORS restrictivo configurable por .env
  - Variables sensibles solo desde .env (godotenv)
  - Manejo centralizado de errores sin exponer detalles internos

✓ BASE DE DATOS:
  - PostgreSQL con pgx/v5 (sin ORM)
  - Todas las queries inyectan SET app.current_user_id para RLS
  - Pool de conexiones optimizado (25 max, 5 min)
  - Índices de rendimiento creados
  - Row-Level Security habilitado

✓ INFRAESTRUCTURA:
  - Redis client (para Pub/Sub futuro)
  - MQTT client (stub para futura integración)
  - Docker Compose con PostgreSQL, Redis y API
  - Dockerfile multietapa optimizado

✓ TOOLING:
  - Makefile con comandos: build, run, test, docker-up/down
  - Scripts de testing con curl
  - Colección Postman lista
  - Script setup.sh para configuración inicial

CÓMO EMPEZAR
============

1. SETUP INICIAL:
   bash setup.sh

2. OPCIÓN A - CON DOCKER (RECOMENDADO):
   docker-compose up -d
   go run ./cmd/main.go

3. OPCIÓN B - LOCALMENTE:
   - Asegúrate que PostgreSQL 16+ corre en localhost:5432
   - Asegúrate que Redis corre en localhost:6379
   - Ejecutar scripts/init.sql en PostgreSQL
   - go run ./cmd/main.go

4. VERIFICAR:
   curl http://localhost:8080/health

5. LOGIN:
   curl -X POST http://localhost:8080/auth/login \
     -H "Content-Type: application/json" \
     -d '{
       "email": "productor@kajve.com",
       "password": "password123"
     }'

6. USAR TOKEN:
   curl -H "Authorization: Bearer <access_token>" \
     http://localhost:8080/lotes
